什么是 Maven

Apache Maven 是一个软件项目管理工具。基于项目对象模型的概念，它可以通过配置信息来管理项目的构建、报告和生成文档这些行为。
其目标旨在：

  • 使项目的构建过程简单
  • 提供一种统一的构建系统
  • 提供优质项目信息
  • 鼓励更好的开发实践

它的重要功能在于项目构建和依赖管理

Maven 的配置有三个层次：

  • Project - 出现在 pom.xml 的静态配置

  • 项目属性 - properties；
  • 构建相关 - build

常常编写和可能改动的基本上是 dependencies 和 build 这两个部分

添加 Maven Wrapper

添加 Maven Wrapper 的目的是方便构建时统一规定 Maven 的版本，还有如果当构建环境没有 Maven 时，也能够构建项目。

1

mvn io.takari:maven:wrapper -DmavenVersion=3.6.3

项目添加了一些文件和目录：

├── mvnw.cmd
├── .mvn
    └── wrapper

        ├── maven-wrapper.jar
        ├── maven-wrapper.properties

Maven 构建生命周期

Maven 构建生命周期（build lifecycle）定义了工件的构建和发布的过程。Maven 的三个内置的生命周期：default、clean 和 site。

  • clean lifecycle：处理项目的清除工作

  • default lifecycle：处理项目的部署

  • site lifecycle：创建项目的站点

构建阶段

每个构建生命周期（build lifecyle）由不同阶段（phases）列表定义，当中一个构建阶段代表的是构建周期的一个层次。

比如 default 生命周期有很多阶段，包括以下多个阶段（这还不是最完整的生命周期的时期列表）：

  • validate -验证项目正确以及所有必要信息可用。
  • compile - 编译项目源代码。
  • test - 拿合适单元测试框架测试编译后的源代码，比如 Junit。这些测试代码不需要打包或部署。
  • package - 取出编译后的源代码打包成发布格式，例如 JAR。
  • verify - 对集成测试结果进行检查，以确保符合质量标准。
  • install - 安装包至本地仓库，以被本地项目依赖使用
  • depoly - 完成构建环境，拷贝最终包到远程仓库，以分享给其他的开发者和项目

以上的构建阶段是按序完成生命周期。当使用 default 生命周期，首先合法性验证项目，接着尝试编译项目源码，运行相应单元（白盒）测试，打包二进制文件（例如 jar），运行包的集成（黑盒）测试，审核集成测试，安装审核后的包到本地仓库，接着部署安装的包到远程仓库。

常用命令行召唤

选择符合结果的阶段。如果想要 jar，运行 package。如果想要运行单元测试，运行 test

如果不确定想要什么，优先调用这个时期：

1

mvn verify

此命令会在执行 verify 时期前按序执行它之前的属于 default 的每个阶段。这个调用会打包二进制文件，还会做集成测试，还有可能会完成额外的检查，比如使用 预定义 checkstyle 规范所写代码。

在构建环境，使用下面的调用来整洁地构建和部署工件到共享仓库

1

mvn clean deploy

这命令在多模块场景下使用，Maven 会传递每个子项目，以及执行 clean，接着执行 depoly（包含之前构建时期的步骤）。

插件目标

一个构建阶段对一个特定的生命周期的步骤负责。实施这责任有许多方式。这责任由绑定了构建阶段的插件目标们来完成。

插件目标（goal）代表助力于构建和管理项目特定任务（更细致的构建阶段）。插件目标可以绑定零个或多个构建阶段。插件目标可在绑定不了任何阶段脱离生命周期直接调用。每个实施（execution）的顺序取决于目标和受调用的构建阶段的顺序。

例如：

1

mvn clean dependency:copy-dependencies package

命令实施时，clean 阶段（包括它之前的 clean生命周期阶段）先实施。接着在实施最后的 package 阶段（含有 default 生命周期它前面的阶段）之前实施 dependency:copy-dependencies 目标

一些不常在命令行召唤的阶段

一些带有连接号阶段名（像 pre-*，post-*或 process-*）不会直接出现在命令行。这些阶段对构建进行排序，产生在构建之外无用的中间结果。例如调用 integration-test，测试环境处于挂起状态。

具体来说，代码覆盖率工具像 Jacoco，实施容器插件像 Tomcat、Cargo 和 Docker 会把目标绑定到 pre-integration-test阶段来准备集成测试容器环境。这些插件还把目标绑定了 post-integration-test阶段来收集测试的统计数据或退出集成测试容器

故障安全工具和代码覆盖率工具把目标绑定到 integration-test 和 verify 阶段。在 verify阶段后最终结果是测试和代码覆盖率报告可用。如果召唤 integration-test ，那将无报告，更坏地是集成测试容器挂起；Tomcat Web 服务器或者 Docker 实例还在运行，Maven 却无法停止。

配置依赖

引入项目所需的依赖包。如何引入？导入依赖的坐标即可，如下描述：

<dependencies>
  ...
  <dependency>
    <groupId>junit</groupId>
    <artifactId>junit</artifactId>
    <version>4.12</version>
    <type>jar</type>
    <scope>test</scope>
    <optional>true</optional>
  </dependency>
  ...
</dependencies>
  • type - 引入依赖包的打包格式，一般忽略不写。如果是 pom，往往 scope 设置为 import，这种依赖只能放在 dependencyManagement下
  • scope - 项目依赖包的作用域，有那么几个：compile、test、provided 和 runtime 还有 import。
  • optional - 表示可选依赖。决定是否向下传递，true 则不传递，false 就传递，默认 false。
作用域 描述
compile 默认的作用域，编译时和运行时都有效
test 测试编译时和运行时有效
provided 编译时有效，运行时有容器提供，比如 javax.servlet:javax.servlet-api
runtime 编译时无效，运行时有效，例如 mysql-connector-java

假如 A 依赖 B，B 依赖 C，那么 A 间接依赖 C

依赖冲突时，发生：

  1. 如果直接与间接依赖中包含有同一个坐标不同版本的资源依赖，以直接依赖的版本为准(就近原则)
  2. 如果直接依赖中包含有同一个坐标不同版本的资源依赖，以配置顺序下方的版本为准(就近原则)

假如项目 X 依赖项目 maven-embedder，maven-embedder 依赖项目 maven-core,但 X 不依赖项目 maven-core。所以在引入 maven-embedder 时，要排除 maven-core。

配置资源文件

指定资源位置，可用于使用代码生成的项目中，比如：mybatis-generator

概况如下：

<build>
    ...
    <resources>
      <resource>
        <targetPath>META-INF/plexus</targetPath>
        <filtering>false</filtering>
        <directory>${basedir}/src/main/plexus</directory>
        <includes>
          <include>configuration.xml</include>
        </includes>
        <excludes>
          <exclude>**/*.properties</exclude>
        </excludes>
      </resource>
    </resources>
    <testResources>
      ...
    </testResources>
    ...
</build>
  • resources - 用于定义资源的列表。
  • targetPath - 资源拷贝到打包文档的目标路径。通常指定为 META-INF，不指定默认从文档的根路径开始。
  • directory - 定义了资源的位置。 构建的默认目录是 ${basedir}/src/main/resources。
  • includes - glob模式匹配文件名，表示要包含的资源文件
  • excludes：与 includes 的结构相同，但指定要忽略的文件。 在包含和排除之间的冲突中，排除优先。
  • testResources：类似于 resources, 针对是测试资源。
